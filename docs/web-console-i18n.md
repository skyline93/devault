# Web 控制台国际化方案（DeVault Console）

本文档定义企业控制台（`console/`：Umi Max 4、Ant Design 5、`@ant-design/pro-components`）的国际化目标、原则与验收口径，供实现与 Code Review 对齐。**实现状态以代码为准**；未落地前本文档为权威需求说明。

---

## 1. 目的与适用范围

**目的**

- 控制台界面支持**英文**与**简体中文**，满足正式环境下的语言偏好。
- **默认使用英文**展示，用户可显式切换为简体中文。
- 方案优先采用 **Ant Design 与 Umi / Ant Design Pro 惯用栈**，避免自研文案框架或重复实现日期、分页等组件级翻译。

**适用范围**

- SPA 内所有面向最终用户的可见文案：布局、菜单、按钮、表单、校验提示、空状态、表格列标题、步骤条、抽屉、通知、页面标题、工具提示等。
- 与 Ant Design / ProComponents 绑定的组件级语言包（分页、日历、弹窗按钮文案等）。
- 建议同步：文档标题（浏览器页签）、根节点 `lang` 属性（可访问性与语义）。

**非目标（可单列后续）**

- 用户文档站（`website/docs/`）的翻译策略。
- 后端错误原文的逐字国际化（见第 7 节：以前端映射与友好文案为主）。

---

## 2. 核心原则

### 2.1 优先 Ant Design 生态，避免重复造轮子

| 层级 | 要求 |
|------|------|
| 组件库文案 | 使用 **Ant Design 官方语言包**（`antd/locale/en_US`、`antd/locale/zh_CN` 等），通过根级 **`ConfigProvider`** 的 **`locale`** 随当前界面语言切换。 |
| 应用文案 | 使用 **Umi Max 内置国际化能力**（与 Ant Design Pro 一致：`src/locales/` 多文件、`useIntl` / `formatMessage` 等），**不**引入与栈并行的第二套 i18n 运行时，除非有明确缺口并经评审。 |
| Pro 布局与表格 | **`@ant-design/pro-components`** 与 `antd` 版本对齐；若 Pro 提供独立 `locale` 或 `ConfigProvider`，与上述语言状态**单一数据源**同步，避免出现 Ant Design 为英文而 Pro 仍为中文等情况。 |
| 日期与时间 | 若使用 **dayjs**（Ant Design 5 默认），在切换语言时同步设置 **dayjs** 的 **`locale`**，与 `antd` 语言包一致。 |

### 2.2 语言不跟随操作系统或浏览器

- **禁止**将 `navigator.language` / `Accept-Language` 作为默认界面语言或静默覆盖用户已选语言。
- **默认语言**固定为 **英文（`en-US` 或项目统一约定的英文 locale id）**。
- 首次访问、无痕模式、清除站点数据后：均显示英文，直至用户在界面内选择其他语言。

### 2.3 可扩展（后续增加语言）

- 文案以 **稳定的 message id**（键）组织，**禁止**以某一种自然语言的句子作为代码中的主键。
- 每种自然语言对应**独立资源模块**（例如 `src/locales/en-US.ts`、`src/locales/zh-CN.ts`），新增语言时：**增加资源文件 + 在配置中注册 locale id + 接入 Ant Design 对应 `antd/locale/*`（若 Ant Design 提供该语言包）**，不在业务组件内写死「仅中英」分支。
- 语言列表（下拉选项）应由**配置或常量表**驱动，避免每增一种语言就改多处 UI。
- 若某语言缺少某条翻译，应有**明确降级策略**（例如回退到英文并可在开发构建时告警），避免界面出现裸 key。

### 2.4 页面严禁暴露内部设计与开发术语

用户可见文案**不得**用于说明实现细节或内部命名，包括但不限于：

- **API / 协议 / 存储字段名**作为标题或主标签（例如 `policy_id`、`idempotency_key`、`FileBackupConfigV1`）；若需「高级配置」，应使用产品化名称（如「备份配置（JSON）」「可选：幂等键」），技术标识仅允许出现在开发文档或受控的「复制诊断信息」类功能中（若产品提供）。
- **内部服务名或拓扑**（例如「IAM」「控制面」「gRPC」「代理端口」）作为面向用户的错误主文案；应改为「身份验证服务不可用」「无法连接到服务」等**不泄露架构**的正式表述，必要时用「联系管理员」收口。
- **仓库、分支、Issue 编号、设计文档段落标记**等出现在任何 UI 字符串中。

### 2.5 表述须专业、正式

- 中英文各自保持**正式语体**，避免口语、网络俚语、过度感叹。
- 同一概念在中英文之间维护**术语对照**（见第 6 节），产品、文档、控制台三处宜逐步对齐。

## 3. 默认语言、切换与持久化

| 项目 | 要求 |
|------|------|
| 默认 | 英文。 |
| 切换入口 | 全局显著位置（例如顶栏用户区附近）：明确展示可选语言（如 English / 简体中文）。 |
| 持久化 | 用户选择写入 **`localStorage`**（或后续与用户配置接口同步）；键名与现有存储键风格一致，避免与 **`devault_tenant_id`**、**`devault_bearer_token`** 冲突。 |
| 切换后 | 当前会话立即重渲染；**Ant Design `ConfigProvider`** 与 **Umi locale** 同源更新。 |
| 根节点 | **`document.documentElement.lang`** 与当前语言一致（如 `en`、`zh-CN`）。 |

---

## 4. 路由、菜单与页面标题

- 路由配置中的 **`name`** 宜为 **message id**（或 Umi locale 插件约定的可翻译键），**禁止**将某一自然语言写死在 `config/routes` 中作为主展示源。
- 侧栏、面包屑等 Pro Layout 展示名称均来自国际化资源，保证与当前语言一致。
- 浏览器标题（若可配置）随语言或路由变化时，使用翻译后的短标题，避免硬编码单一语言。

---

## 5. 文案键与文件组织（建议）

- 键名：分层前缀，例如 `menu.*`、`page.backup.*`、`error.network.*`，便于检索与去重。
- 插值变量命名语义化（如 `tenantName`），注意中英文语序差异（必要时拆成多条 id 或使用 ICU 风格由 `formatMessage` 支持）。
- 全局错误与请求拦截器（如 `requestErrorConfig`）中的提示须全部改为 **id + 多语言资源**，且符合第 2.4、2.5 节。

---

## 6. 术语表（维护要求）

- 在仓库内维护一份**控制台专用术语表**（可与本文件同目录或 `console/` 下，格式 CSV / Markdown 均可），列出核心概念的**英文正式用语**与**简体中文正式用语**对照。
- 新增业务概念时：先更新术语表，再添加 message id，避免同一英文概念在中文界面出现多种译法。

---

## 7. 与后端及网络错误的关系

- 优先展示**前端映射**后的友好文案；对后端返回的自由文本 **`detail`** 应谨慎直出，防止泄露堆栈、路径或内部服务名。
- 若后端提供稳定 **`code`** 或分类，建议建立 **code → message id** 映射表；未知错误使用通用兜底文案（中英各一条）。

---

## 8. 验收标准（建议）

1. 清除站点数据后首次打开：界面为**英文**，Ant Design 组件（日期、分页等）为英文。
2. 切换为简体中文后：界面与 Ant Design 组件为中文；刷新页面后**保持**简体中文。
3. 不出现裸 message id；不出现未翻译的回退异常（除非已文档化的降级策略）。
4. 抽检主要路径（登录、租户切换、备份作业、策略、发起备份等）：**无**第 2.4 节禁止类内部术语作为主文案。
5. 新增一种语言（演练）：仅增加资源文件与配置注册即可使新语言出现在切换列表并加载（不要求一次性译满，但不得破坏构建）。

---

## 9. 与当前实现的差距说明（基线）

当前 `console/config/config.ts` 中将 **Ant Design** 全局 **`locale`** 固定为 **`zh_CN`**，且路由 **`name`** 多为中文常量；各页面存在大量硬编码中文及部分技术向用语。落地本方案时需逐项迁移至 **Umi locale + Ant Design 语言包联动**，并改写不符合第 2.4、2.5 节的文案。
